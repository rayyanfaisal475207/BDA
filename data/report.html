Project Overview

Our team developed Flowra to tackle a major headache in Agile software development: the gap between what's happening in the code and what the Jira board actually shows. Most developers find it a chore to manually update tickets, so the board ends up being out of sync. Flowra solves this by listening to team chat (like Discord or Slack) for status updates, verifying that work against real GitHub commits, and then asking the PM for a quick approval to move the card. This report breaks down our research, the gaps we found in existing tools, and the full set of requirements we've defined for the project.

System Scope and Definition

The Challenges We're Tackling

Problem 1: The Manual Update Chore

In the middle of a sprint, developers often forget to drag their Jira cards. The friction of switching from coding to a browser just to update a ticket creates a "lagging reality" where the board doesn't match the actual codebase. We're removing this manual step.

Problem 2: Automation without Evidence

Current automation is too simple it just looks at branch names. It doesn't actually check if the code address the ticket. Flowra adds a verification layer to make sure a task is truly finished before suggesting a move.

Problem 3: The "Invisible" Contributors

Tools like LinearB are great for coders but ignore QA testers and PMs. Since they aren't pushing code to GitHub, their effort often goes unmeasured. Flowra tracks signals from every role to give a fairer view of team performance.

Our Integrated Solution

Flowra fixes these issues by bridging the gap between how a team talks, how they code, and how they manage their board. It creates a single, verified pipeline that keeps everyone in sync without the manual overhead.

2.3.1 Our Goal with This Research

We wanted to see how existing tools handle Agile workflows and where they fail. We looked at four popular systems that each tackle the problem differently: Jira (the classic), LinearB (for metrics), Zapier (for DIY automation), and Notion AI (for documentation). Our goal was to find exactly what's missing in the market so we could build Flowra to fill those gaps.

2.3.2 How We Approached the Analysis

We reviewed official documentation, product feature pages, and publicly available workflow guides for each system. We evaluated each system against six questions that are directly relevant to what Flowra is trying to do:

  • Does it move Jira cards automatically based on real developer activity, or does someone still have to do it manually?
  • Does it verify that the work was actually done before moving a card, or does it trust a branch name and call it done?
  • Can it listen to Discord, Slack, and Telegram at the same time and understand what is being said?
  • Is there a step where a human reviews and approves a suggested change before it hits Jira?
  • Does it track the performance of non-developer roles like marketing, QA, or project management?
  • How much effort does it realistically take to set up and maintain for a small team?

2.3.3 Detailed Audit Findings

Tool A: Jira Software

Jira is pretty much the industry standard, and for good reason. It handles the basics like Kanban boards and sprint tracking perfectly. We really liked its built-in automation engine because it's so flexible you can tell it to move a card when a PR is opened, which is a great start.

What we found: While the automation is powerful, it's also "blind." It doesn't actually understand if the code in the PR is any good; it just sees an event and reacts. This leads to the board getting cluttered with "done" cards that aren't actually finished.

Tool B: LinearB

LinearB is great for engineering managers who love data. It pulls metrics like cycle time and PR throughput straight from Git, which is awesome for seeing how fast a team is moving.

What we found: The biggest issue we saw is that it completely ignores anyone who isn't writing code. If you're a QA tester or a PM, you basically don't exist in LinearB's eyes. Also, it doesn't "listen" to the team chat where most of the decisions actually happen.

Tool C: Zapier + GitHub + Slack

We looked at this as a DIY alternative. You can basically stitch together your own automation using webhooks between GitHub and Slack.

What we found: It's just too fragile. If a developer changes a branch naming convention, the whole thing breaks. Plus, you have to build and maintain every single integration yourself, which is a massive headache for a small team.

Tool D: Notion AI

Notion has been adding a lot of AI features lately, mostly for summarizing notes and drafting documents.

What we found: It's great for documentation, but it's not a workflow tool. It has no idea what's happening in Git or Jira. It can summarize a meeting, but it can't actually move the project forward based on real work evidence.

2.3.5 The Five Gaps We're Filling

1. Chat is a "Black Hole"

Teams talk about work all day on Discord and Slack, but none of that info ever makes it to Jira automatically. We're turning those "I'm done" messages into real triggers.

2. Blind Automation

Most tools move cards just because a branch was created. They don't actually understand the code. Flowra double-checks the GitHub activity to make sure the work was actually done.

3. Lack of Human Oversight

Current automation often messes up the board because it happens too fast without a check. We've built an "Approval-First" model so the PM stays in control.

4. Non-Coders are Left Out

If you aren't writing code, current tools basically ignore you. We're tracking metrics for QA and PM roles so everyone's contribution is visible.

5. Platform Fragmentation

Switching between Slack, GitHub, and Jira is where the friction happens. Flowra bridges all these platforms into one unified workflow.

2.3.6 Industrial Synthesis

Looking at everything together, Flowra is trying to fill a gap that none of these tools address: the connection between what a team communicates, what they actually produce in code, and what their project board reflects. Keeping OpenClaw, the Flowra Engine, and the Web App as separate joined layers is the only practical way to build something that can actually do all of these things reliably.

7.1 Stakeholder Discovery Interview Meeting Minutes

Project: Flowra, Agentic Agile Date: April 21, 2026
Participants: Faizan Anwar (Lead), Tanguy De Branbandre (Stakeholder) Duration: 36 Minutes
1. Agenda
  • Validation of industry pain points regarding manual Jira updates.
  • Feasibility review of AI-driven "Proof of Work" verification.
  • Evaluation of the "Approval-First Sync" workflow.
  • Identification of innovative edges (Risk Analysis vs. Automation).
2. Discussion Highlights & Insights

Validating the "Lazy Dev" Problem: Tanguy laughed when we brought up sloppy reporting, but he got serious about the "human in the loop" issue. He made a great point that managers are getting buried under verification work as AI generates more code, which really validated why we need Flowra.

The Security Pivot: Interestingly, Tanguy wasn't as impressed by the simple automation as he was by the Risk Analysis idea. He pushed us to think about Flowra as an audit layer something that catches security leaks (like the public bucket example he mentioned) before they hit production.

Simplifying the Scope: We had a bit of a reality check when discussing generic environments (like construction). Tanguy helped us realize that we should stick to what we know the developer space but build it modularly so other industries could plug in later if they wanted to.

7.2 Team Internal Meeting Minutes (SRS Finalization)

Meeting: Phase 1 Synthesis & SRS Review Date: April 21, 2026
Participants: Faizan, Waleed, Zarsham, Haleema, Anas, Furqan Location: Discord Voice Channel
1. Agenda
  • Review of Stakeholder Interview findings and required SRS pivots.
  • Finalization of MoSCoW prioritization for the Phase 1 Backlog.
  • UML Diagram consistency check (Sequence vs. Class vs. Architecture).

2. Discussion & Decisions

Refining the Backlog: After the interview, we had a long huddle about our priorities. We originally had "Risk Analysis" as a secondary feature, but we decided to bump it up to 'Must Have' because it's our biggest differentiator. We also officially scrapped the "Construction Site" idea to keep our focus sharp on the MVP.

Finalizing the Visuals: Faizan walked us through the final UML diagrams. We spent some time making sure the Sequence Diagram actually matched our Class attributes. We also did a final walkthrough of the SRS to make sure the "Modular Core" message was consistent throughout the document.
