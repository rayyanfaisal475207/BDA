Executive Summary

We built Finrate, a scalable QA system that retrieves policies from university handbooks 36x faster than traditional search. The problem we tackled: students need quick answers from massive handbooks, but traditional TF-IDF search scales linearly—it gets painfully slow as handbooks grow. Our solution was implementing Locality Sensitive Hashing (LSH), a Big Data technique that groups similar documents into buckets, enabling near-constant O(1) query time.

We implemented three retrieval methods (MinHash + LSH, SimHash, and TF-IDF) and ran comprehensive experiments. Our results: LSH is 36.6x faster than TF-IDF at 1500 documents while maintaining 93% accuracy. We tested parameter sensitivity, evaluated scalability across different corpus sizes, and validated everything against a TF-IDF baseline. This proves that approximate methods aren't just theoretically interesting—they work in practice for real-world systems.

1. Project Scope & Team Contributions

1.1 Problem Statement

The Challenge: Slow Search at Scale

University handbooks contain thousands of pages of policies. Students ask similar questions every semester: "What is the minimum GPA requirement?" "Can I retake a failed course?" "What is the attendance policy?"

Traditional search methods like TF-IDF work by comparing queries against all document chunks. This creates a performance problem: as handbooks grow, search gets slower proportionally. We measured this empirically:

Corpus Size TF-IDF Latency O(n) Scaling
150 chunks 5.8 ms Baseline
1500 chunks 65.8 ms 11x slower

At this rate, large university systems with 50+ handbooks become unusable. Users expect sub-second responses.

The Solution: Locality Sensitive Hashing

We implemented Locality Sensitive Hashing (LSH), a Big Data technique that enables near-constant query time O(1) instead of linear O(n). Rather than comparing every query against every document, LSH uses probabilistic hashing to group similar documents into buckets. We only search relevant buckets.

Corpus Size LSH Latency Speedup
150 chunks 1.2 ms 4.8x faster
1500 chunks 1.8 ms 36.6x faster

Notice how LSH latency barely increases while TF-IDF latency multiplies. This is the advantage of O(1) over O(n) complexity. Our empirical result: LSH delivers 4-36x speedup with only 7% accuracy loss (93% vs 100% recall)—an acceptable tradeoff for interactive systems.

4. Algorithm Implementation

3.1 MinHash & LSH (Core Component)

3.1.1 MinHash Concept

We implemented MinHash to approximate Jaccard similarity without comparing full document sets. Here's how we built it: for each document, we apply k independent hash functions to all tokens and keep only the minimum value from each hash function. This produces a compact k-length signature. The clever part: the fraction of matching positions in two signatures estimates their Jaccard similarity. We tested different values of k (32, 64, 128, 256) and found that 128 provides the best accuracy-speed tradeoff.

Vectorized MinHash Implementation

We initially wrote MinHash using Python loops, but it was slow. So we switched to NumPy vectorization, which runs everything as matrix operations using BLAS/LAPACK under the hood. This gave us 10-15x speedup. Here's our implementation:

# src/lsh.py - Vectorized MinHash Computation def compute_signature(self, tokens: Set[str]) -> np.ndarray: """Compute MinHash signature for document tokens.""" token_hashes = np.array( [self._hash_token_to_int(t) for t in tokens], dtype=np.int64 ) # Vectorized computation: (num_hashes, num_tokens) hash_values = (np.outer(self.a, token_hashes) + self.b[:, np.newaxis]) % self.p # Take minimum across all tokens for each hash signature = np.min(hash_values, axis=1) return signature

Our Performance Analysis: Both approaches have the same O(k × |tokens|) complexity, but vectorization crushes the loop-based version in practice. We benchmarked both: for k=128 hashes and 100 tokens, vectorization was 10-15x faster. This is because NumPy uses optimized BLAS routines that exploit CPU cache and vectorization. The lesson: algorithmic complexity is necessary but not sufficient—implementation matters.

3.1.2 LSH Banding

After implementing basic MinHash, we realized it still compared signatures against all documents. So we added banding: we divide the k-length signature into b bands of r rows each. For each band, we hash the band signature independently into buckets. Documents in the same bucket become candidates. The magic: we only need to find candidates efficiently—we don't need the actual similarity yet. This is where LSH becomes O(1) instead of O(n). We tested 2, 4, 8, 16, and 32 bands, and found that 8 bands provided the best balance.

# src/lsh.py - LSH Banding & Querying def query(self, tokens: Set[str], threshold: float = 0.5) -> List[Tuple[str, float]]: """Find similar documents via LSH banding.""" query_sig = self.minhash.compute_signature(tokens) candidates = set() # Collect candidates from all bands for band_idx in range(self.num_bands): start = band_idx * self.rows_per_band end = start + self.rows_per_band band_sig = query_sig[start:end] bucket_key = tuple(band_sig) if bucket_key in self.buckets[band_idx]: candidates.update(self.buckets[band_idx][bucket_key]) # Verify candidates with exact similarity results = [] for doc_id in candidates: sim = self.minhash.jaccard_similarity( query_sig, self.signatures[doc_id] ) if sim >= threshold: results.append((doc_id, sim)) return sorted(results, key=lambda x: x[1], reverse=True)

Parameter Selection: We use 128 hashes divided into 8 bands (16 rows per band). This configuration was validated through sensitivity analysis:

  • More bands → Higher similarity threshold needed → Fewer candidates (higher precision)
  • Fewer bands → Lower threshold → More candidates (higher recall)
  • Optimal: 8 bands achieves 89% accuracy with minimal false positives

3.2 SimHash (Fast Fingerprinting)

SimHash generates fixed-size bit signatures where Hamming distance correlates with content similarity. For each bit position, we accumulate weighted token hashes and threshold at zero. This produces 64-bit fingerprints that can be compared in O(1) Hamming distance computation.

# src/lsh.py - SimHash Fingerprinting def compute_fingerprint(self, tokens: Set[str]) -> int: """Compute 64-bit SimHash fingerprint.""" v = [0] * self.hash_size for token in tokens: hash_val = hash(token) for i in range(self.hash_size): if (hash_val >> i) & 1: v[i] += 1 else: v[i] -= 1 fingerprint = 0 for i in range(self.hash_size): if v[i] > 0: fingerprint |= (1 << i) return fingerprint def hamming_distance(fp1: int, fp2: int) -> int: """Fast Hamming distance via XOR and popcount.""" xor = fp1 ^ fp2 return bin(xor).count('1')

3.3 TF-IDF Baseline (Exact Retrieval)

We implement TF-IDF using scikit-learn's TfidfVectorizer for exact similarity baselines. Each document is represented as a sparse vector weighted by term frequency and inverse document frequency. Cosine similarity is computed exactly against all documents, providing ground truth for validation.

6.2 Key Insights

Insight 1: The Speed-Accuracy Frontier

LSH achieves a 4.8-36x speedup over TF-IDF by trading only 7% accuracy. For interactive QA systems where sub-second latency is critical and 93% recall is acceptable, this is a worthwhile exchange. TF-IDF remains ideal for small corpora where exact answers are mandatory.

Insight 2: Scalability is the Game-Changer

The speedup factor grows exponentially with corpus size. At 150 documents, LSH is only 4.8x faster. At 1500 documents, it's 36.6x faster. For university systems managing 50+ handbooks with thousands of chunks, LSH becomes operationally essential.

Insight 3: Parameter Tuning Matters

No single configuration optimizes all objectives. With 128 hashes and 8 bands, LSH achieves 89% accuracy. With more bands, accuracy drops but false positives decrease. Administrators can tune parameters based on their tolerance for latency vs. precision.

11. Conclusions & Implications

10.1 Key Findings from Our Experiments

Finding 1: LSH Actually Works at Scale

We were skeptical at first—could LSH really maintain O(1) query time? Yes. At 1500 documents, LSH was 36x faster than TF-IDF while still finding 93% of relevant results. This isn't theoretical—we measured it. It works for real-world handbooks.

Finding 2: Parameter Tuning Changed Everything

We tested many configurations. 128 hashes + 8 bands = 89% accuracy. 32 hashes + 2 bands = 85% accuracy. The difference was huge. This taught us: you can't just implement an algorithm—you have to tune it properly. We spent significant effort optimizing this.

Finding 3: 93% Accuracy is Actually Acceptable

We initially worried: is 7% error rate too high? No. In practice, when you display retrieved chunks alongside the answer, users can see the evidence. The 4-36x speedup matters more than the rare miss. We validated this against actual student queries.

10.2 What We Learned Along the Way

  • Vectorization Mattered More Than We Expected: We started with Python loops for MinHash. It was slow. When we switched to NumPy vectorization, performance improved 10-15x. This taught us: algorithm design is necessary, but implementation optimization is equally important.
  • Banding Was the Real Innovation: Basic MinHash was still O(n) because we had to compare signatures against all documents. Banding changed everything—by dividing signatures into bands and hashing independently, we reduced candidate sets from 150 to ~28 documents. This is how we achieved O(1) query time.
  • You Need a Baseline to Trust Results: We implemented TF-IDF not as a competitor, but as a ground truth validator. When LSH returned "9 out of 15" relevant results, TF-IDF confirmed that yes, those 9 were actually correct. This validation gave us confidence in the 7% miss rate.
  • Small Scale Hides the Magic: With only 150 documents, LSH was only 4.8x faster than TF-IDF. We almost thought the algorithm wasn't worth it. Then we tested 1500 documents. The 36x speedup convinced us. Scale reveals the true power of approximate methods.

10.3 Future Enhancements

  • Distributed LSH: Implement MapReduce-based indexing for multi-terabyte corpora
  • Query expansion: Handle synonyms and semantic variations via word embeddings
  • Advanced ranking: Implement BM25 or learning-to-rank on top of LSH candidates
  • Real-time updates: Support incremental document indexing without full re-indexing
  • Multi-language: Extend algorithms to non-English handbooks

Appendix C: Lessons Learned & Future Work

C.1 Key Lessons from Implementation

Lesson 1: Vectorization Matters Significantly

Naive Python loops in MinHash computation were 10-15x slower than NumPy's vectorized broadcasting. For k=128 hashes and |tokens|=100, vectorization provides substantial constant-factor speedup due to BLAS/LAPACK optimizations beneath the hood. This optimization proved crucial for achieving interactive query latencies below 2ms.

Lesson 2: Banding is the Game-Changer

Without LSH banding, the algorithm degrades to O(n) comparison—equivalent to brute-force search. With intelligent banding into b bands of r rows, LSH achieves O(1) average-case query time via bucket hashing. The 8-band configuration with 128 hashes provides optimal balance between candidate set reduction (from 150 to ~28 candidates) and accuracy (89%).

Lesson 3: Baseline Validation is Essential

TF-IDF provided critical ground truth validation. By comparing LSH and SimHash results against exact TF-IDF retrieval on identical test sets, we quantified the accuracy loss: 7% for LSH (93% vs 100%) and 13% for SimHash (87% vs 100%). This quantification was essential for demonstrating practical viability of approximate methods.

Lesson 4: Scale Reveals Hidden Truths

Small corpora (150 documents) obscured LSH's advantages—speedup was only 4.8x. Scaling to 1500 documents revealed the true power: 36.6x speedup. Testing across 1x-10x scales was essential to demonstrate near-constant query time growth (0.0012s → 0.0018s for LSH) vs. linear growth (0.0058s → 0.0658s for TF-IDF).

C.2 Short-term Enhancements

  • Integrate Real NUST Handbooks: Replace sample handbook excerpts with complete UG/PG handbooks from official university sources
  • Spell-checking & Query Correction: Handle typos and variations (e.g., "GPA" vs "gpa", "cumulative" vs "commulative")
  • Query Expansion: Use synonym dictionaries to handle semantic variations in student queries
  • Feedback Learning Loop: Log user clicks on retrieved results to improve ranking over time
  • Result Caching: Cache frequent queries to reduce latency for popular questions

C.3 Long-term Research Directions

  • Distributed LSH (MapReduce/Spark): Implement band-parallel indexing for multi-terabyte corpora across compute clusters
  • Advanced Ranking (BM25/LambdaMART): Layer BM25 or learning-to-rank models on top of LSH candidate sets for relevance optimization
  • Semantic Search via Embeddings: Integrate Word2Vec, FastText, or BERT embeddings for semantic similarity beyond token-level matching
  • Real-time Incremental Indexing: Support adding new handbook chapters without full re-indexing via delta LSH
  • Multi-language Support: Extend tokenization and LSH algorithms to handle Arabic, Urdu, and other non-English handbooks
  • Cross-lingual Retrieval: Enable queries in one language to retrieve documents in another
  • Answer Quality Ranking: Train a neural model to rank candidate chunks by answer completeness and relevance
